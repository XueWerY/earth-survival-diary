# Earth Survival Diary 项目开发技能

此技能用于 `earth-survival-diary`（地球 Online 生存日记）项目的所有开发任务。当用户提出任何需要修改该项目代码的需求时，调用此技能。

---

## 一、数据持久化规则

所有模块的数据统一从 `C:\Users\<用户>\AppData\Roaming\earth-survival-diary\data` 存取：

| 模块 | 存储路径 |
|------|----------|
| 账户信息 | `data\users` |
| 用户数据（根） | `data\<用户ID>\` |
| 倒数日模块 | `data\<用户ID>\countdown` |
| 课程表模块 | `data\<用户ID>\course` |
| 专注模块 | `data\<用户ID>\focus` |
| 足迹模块 | `data\<用户ID>\footprint` |
| 清单模块 | `data\<用户ID>\list` |
| 笔记模块 | `data\<用户ID>\note` |
| 我的模块 | `data\<用户ID>\profile` |
| 会话数据 | `data\<用户ID>\session` |
| 设置数据 | `data\<用户ID>\settings` |
| 系统状态 | `data\<用户ID>\system` |

**实现要求**：
- 数据持久化使用本地 JSON 文件，同时更新导入/导出/清空数据功能逻辑。
- 确保程序启动后统一加载所有功能页面的数据，切换到各个功能页面后不再重复加载数据。

## 二、导航栏修改规则

当任务涉及修改导航栏时，必须满足以下要求：

1. **响应式布局**：当导航栏宽度小于窗口宽度时，导航栏水平居中；当导航栏宽度大于窗口宽度时，通过左右滑动查看导航项。
2. **隐藏滚动条**：滑动状态下不显示滚动条（使用 CSS 隐藏）。
3. **平滑居中**：选中导航项或改变窗口尺寸时，实现平滑水平居中过渡（使用 `scrollIntoView({ behavior: 'smooth', block: 'nearest', inline: 'center' })` 或 `scrollTo({ left, behavior: 'smooth' })`）。

## 三、编码规范

### 3.1 写前先读

在修改任何文件之前，必须先用 Read 工具完整阅读该文件，理解其现有逻辑、代码风格和依赖关系。禁止在不了解文件内容的情况下直接修改。

### 3.2 日志记录

根据实际情况，在修改的代码中添加详细的日志记录。使用项目已有的日志工具 [logger.ts](file:///c:/Project/earth-survival-diary/src/lib/logger.ts)。

### 3.3 删除二次确认

任何涉及删除操作的代码，必须添加二次确认弹窗，防止用户误操作。使用项目已有的 [ConfirmDialog.vue](file:///c:/Project/earth-survival-diary/src/components/ConfirmDialog.vue) 或 Element Plus 的 `ElMessageBox.confirm`。

### 3.4 数据统一加载

程序启动后统一加载所有功能页面的数据（在路由守卫或 App.vue 的 onMounted 中集中处理），各功能页面切换时不再重复加载数据。如需刷新，提供手动刷新机制。

## 四、思考原则

在开始编码前，必须遵循以下原则：

### 4.1 编码先思考

- 清晰列明所有假设，存在疑问就主动确认。
- 若需求存在多种解读方式，全部列出，不私自选定一种。
- 若存在更简洁的实现方案，主动提出；必要时敢于提出异议。
- 若内容存在模糊不清之处，暂停开发，明确指出疑问点并询问。

### 4.2 优先简洁至上

- 用最少的代码解决问题，不编写任何多余臆测内容。
- 不开发需求以外的额外功能。
- 仅单次使用的代码，不做抽象封装。
- 不额外实现未要求的扩展性、可配置性功能。
- 不对不可能出现的场景编写异常处理逻辑。
- 若代码可以大幅精简，必须重构精简。

### 4.3 精准修改代码

- 只改动必要内容，仅清理自身修改产生的冗余。
- 不擅自优化周边代码、注释、代码格式。
- 不对运行正常的代码进行重构。
- 遵循原有代码风格（缩进、命名、导入方式等），即便有更好的编写方式。
- 若发现无关的无用代码，仅备注说明，不擅自删除。
- 删除本次修改后不再使用的导入语句、变量、函数。但未经要求，不删除原本就存在的无用代码。

### 4.4 目标导向执行

将任务转化为可验证的目标，完成后进行验证：

- "新增校验逻辑" → 先编写非法入参测试用例，再实现代码使用例通过。
- "修复漏洞" → 先编写可复现漏洞的测试用例，再修复问题使用例通过。
- "重构模块" → 保证重构前后所有测试用例均正常通过。

对于多步骤任务，列出执行计划：
- 一、[步骤] → 验证方式：[内容]
- 二、[步骤] → 验证方式：[内容]

## 五、任务完成前必做清单

每次代码修改完成并通过验证后，必须依次执行以下四项：

### 5.1 更新版本号

在 [package.json](file:///c:/Project/earth-survival-diary/package.json) 中更新 `version` 字段。版本号格式：`YYYY.MM.DD-X`，与今日日期对应。X 为当日第 X 次发布（从 1 开始递增）。例如：`2026.05.18-1`。

若当天已有版本记录，在已有最大 X 基础上 +1。

### 5.2 更新 README.md

在 [README.md](file:///c:/Project/earth-survival-diary/README.md) 中更新项目说明，反映本次修改的内容。

### 5.3 更新 CHANGELOG.md

在 [CHANGELOG.md](file:///c:/Project/earth-survival-diary/CHANGELOG.md) 文件顶部（标题行下方）添加更新日志条目。格式：

```markdown
### v<版本号> (<日期>)
- <用非程序员也能看懂的语言描述修改内容>
```

### 5.4 更新开发技术文档

同步更新 `docs/` 目录下受影响的 VitePress 技术文档。文档位于 [docs/](file:///c:/Project/earth-survival-diary/docs/) 目录，包含项目简介、技术栈、架构、核心功能、数据存储等章节。仅更新与本次修改相关的文档页面。
