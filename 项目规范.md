# Earth Survival Diary 项目开发规范

---

## 一、项目概述

### 1.1 项目简介

**Earth Survival Diary（地球 Online 生存日记）** 是一款帮助用户记录生活足迹、管理任务清单、追踪专注时间的桌面与移动端效率应用。项目采用前后端一体化架构，前端基于 Vue 3 + TypeScript 构建，后端内嵌 Express 服务，通过 Electron 封装为 Windows 桌面应用，通过 Capacitor 打包为 Android 移动应用。

### 1.2 平台支持

| 平台    | 技术方案                     | 数据存储                                   |
| ------- | ---------------------------- | ------------------------------------------ |
| Windows | Electron（NSIS 安装包）      | `%APPDATA%/earth-survival-diary/data/`    |
| Android | Capacitor（APK）             | Capacitor Filesystem（本地 JSON 文件）     |

- 桌面端默认窗口大小：全屏（根据屏幕分辨率），禁用了最大化按钮和拖拽调整大小，可在「我的 → 系统设置 → 窗口分辨率」中切换。分辨率选项以物理像素展示（乘以 Windows 缩放因子如 125%），选择最大分辨率时窗口进入全屏无边框模式，选择较小分辨率时窗口退出全屏并居中显示
- 桌面端更新或重装不会影响已有数据，正常卸载时默认保留用户数据（`deleteAppDataOnUninstall: false`）；如需彻底清除，可手动运行 `scripts/uninstall-old-version.bat` 脚本，该脚本提供交互式数据删除选项
- 移动端卸载应用会清除本地数据
- **安卓端全屏显示**：应用界面延伸至系统状态栏和导航栏区域。通过 `MainActivity.java` 中 `WindowCompat.setDecorFitsSystemWindows(window, false)` + 透明状态栏/导航栏实现 edge-to-edge 渲染（注意：不可添加 `FLAG_DRAWS_SYSTEM_BAR_BACKGROUNDS`，该 flag 与 `setDecorFitsSystemWindows` 冲突会导致部分设备全屏失效）；前端通过 `viewport-fit=cover` + CSS `env(safe-area-inset-*)` 变量自动适配内容区。`index.html` 的 viewport meta 必须包含 `viewport-fit=cover`，`:root` 中定义 `--safe-area-inset-top` 和 `--safe-area-inset-bottom` CSS 变量，`.app-container` 需同时设置 `padding-top` 和 `padding-bottom` 使用对应安全区变量，确保全局导航栏位于系统导航栏上方而非被遮挡

### 1.3 功能模块概览

| 模块       | 路由路径       | 核心功能                                                   |
| ---------- | -------------- | ---------------------------------------------------------- |
| 足迹       | `/footprint`   | 活动记录、日记、时间线视图、公历/农历双历法                  |
| 专注       | `/focus`       | 番茄钟/正计时、星轨视觉动画、常用专注                        |
| 清单       | `/list`     | 清单/分组/任务管理、周期性任务、优先级、提醒                 |
| 倒数日     | `/countdown`   | 倒计时/正计时管理、农历日期、节日自动生成、分类管理          |
| 课程表     | `/course`      | 周视图网格、多天/多节次课程、智能节次计算                    |
| 统计       | `/statistics`  | 五合一统计面板、趋势图、日期范围筛选                        |
| 工具箱     | `/toolbox`     | 插件小工具集合、文件管理（含数据导出导入）             |
| 我的       | `/profile`     | 个人资料、设置、关于、新手引导                              |

## 二、开发环境配置

### 2.1 环境要求

| 依赖        | 最低版本    | 说明                       |
| ----------- | ----------- | -------------------------- |
| Node.js     | ≥ 18        | 运行时环境                 |
| pnpm        | 最新稳定版  | 包管理器（**必须使用 pnpm**） |
| Android Studio | 最新稳定版 | Android 构建（仅移动端开发） |

### 2.2 依赖安装

```bash
pnpm install
```

### 2.3 开发命令

| 命令                         | 说明                              |
| ---------------------------- | --------------------------------- |
| `pnpm build`                 | 构建 Web 资源（Vite 构建到 `dist/`） |
| `pnpm electron:build:win`    | 构建 Windows 安装包（完整流程）    |
| `pnpm publish-release`       | 查看发布指引                      |

**构建流程说明**（`electron:build:win` 依次执行）：
1. `pre-clean-release` — 清空 `release/` 目录
2. `build` — Vite 构建前端资源
3. `install-deps` — 安装 Electron 生产依赖（Express、CORS）
4. `generate-icons` — 从 `build/app-icon.png` 生成所有平台图标（桌面端 PNG/ICO + Android mipmap）
5. `make-ico` — 生成多尺寸 ICO 图标
6. `electron-builder --win` — 打包 NSIS 安装程序
7. `fix-latest-yml` — 修正 `latest.yml` 中的下载 URL
8. `cleanup-release` — 清理构建中间产物

**Android 构建**（在 Android Studio 中操作）：
```bash
node scripts/generate-icons.js       # 生成图标（含 Android mipmap）
pnpm build && npx cap sync           # 构建并同步 Web 资源
npx cap open android                 # 在 Android Studio 中打开项目
# 在 Android Studio 中：Build → Build Bundle(s) / APK(s) → Build APK(s)
```

Android 端的版本号（`versionName`）在 `android/app/build.gradle` 中通过 Groovy 脚本自动从 `package.json` 的 `version` 字段读取，无需手动同步。

### 2.4 IDE 配置

- 推荐使用 **VS Code** 开发
- 必须安装 **Vue - Official（Volar）** 插件，禁用 Vetur
- TypeScript 配置文件：
  - `tsconfig.json` — 项目引用根配置
  - `tsconfig.app.json` — 应用源码配置（`strict: true`）
  - `tsconfig.node.json` — 构建脚本配置

### 3.3 新增文件/组件的目录选择规则

按以下优先级判断新文件的目标目录：

1. **属于某个功能模块**（足迹/专注/清单/倒数日/课程表/统计/工具箱/我的）→ 放入对应 `components/<模块>/` 子目录
2. **属于认证流程** → 放入 `components/auth/`
3. **属于跨模块通用弹窗/遮罩** → 放入 `components/common/overlay/`
4. **属于跨模块通用选择器** → 放入 `components/common/picker/`
5. **属于跨模块通用卡片类组件** → 放入 `components/common/card/`
6. **可复用的组合式函数** → 放入 `composables/`
7. **纯工具/服务类** → 放入 `lib/`
8. **业务状态管理** → 放入 `stores/`（Pinia Store）
9. **仅单模块使用且无复用可能** → 就近放入对应模块目录，**不放入 common**

### 4.3 状态管理

#### Store 设计模式

所有 Pinia Store 统一使用 **Setup Store（Composition API）** 风格，遵循以下标准模式：

```typescript
export const useXxxStore = defineStore('xxx', () => {
  const data = ref<Xxx[]>([])
  const isLoaded = ref(false)

  const loadData = async () => {
    if (isLoaded.value) return
    // ... 加载逻辑
    isLoaded.value = true
  }

  const reset = () => {
    data.value = []
    isLoaded.value = false
  }

  return { data, isLoaded, loadData, reset }
})
```

#### Store 职责划分

| Store             | 职责                                  | 数据来源            |
| ----------------- | ------------------------------------- | ------------------- |
| `authStore`       | 用户认证、个人信息、登录/注册/登出     | `api.ts` + `storageService` |
| `taskStore`       | 足迹记录增删改查                       | `api.ts`            |
| `focusStore`      | 专注记录、常用专注、计时状态、统计计算  | `storageService`    |
| `listStore`    | 清单/分组/任务管理、重复策略常量       | `api.ts`            |
| `settingsStore`   | 专注设置、课程表设置、提醒设置         | `api.ts`            |

#### 命名约定

| 操作       | 方法命名模式        | 示例                    |
| ---------- | ------------------- | ----------------------- |
| 获取数据   | `loadXxx()` / `getXxx()` | `loadData()`, `getTasks()` |
| 添加       | `addXxx()`          | `addTask()`             |
| 更新       | `updateXxx()`       | `updateTask()`       |
| 删除       | `deleteXxx()`       | `deleteCountdown()`     |
| 切换状态   | `toggleXxx()`       | `toggleStar()`          |
| 重置       | `reset()`           | `reset()` （登出时清理） |

### 4.4 数据持久化

#### 存储模式

项目根据运行环境自动选择存储模式：

| 模式           | 环境判断                                 | 存储方式                                                             |
| -------------- | ---------------------------------------- | -------------------------------------------------------------------- |
| Electron 模式   | `window.electronAPI` 存在                 | 通过 Express 后端 API 读写 `%APPDATA%/earth-survival-diary/data/` 下 JSON 文件 |
| Capacitor 模式  | `window.electronAPI` 不存在               | 通过 `fileStore.ts` 使用 Capacitor Filesystem 插件读写本地 JSON 文件   |

判断逻辑：
```typescript
const isCapacitor = typeof window !== 'undefined' && !(window as any).electronAPI
```

#### 数据存储路径

Electron 模式下，所有模块数据通过 Express 后端存取，实际路径如下：

| 模块       | 存储路径                                                               |
| ---------- | ---------------------------------------------------------------------- |
| 全局设置   | `data/settings/settings.json`（非用户隔离，如自动填充密码）              |
| 账户信息   | `data/users/<邮箱>.json`                                               |
| 足迹       | `data/<用户ID>/footprint/footprint.json`                                |
| 专注       | `data/<用户ID>/focus/favorites.json`、`records.json`                   |
| 清单       | `data/<用户ID>/list/lists.json`、`tasks.json`（含 repeat_lunar_month / repeat_lunar_day 农历重复字段）、`folders.json`、`favorites.json`           |
| 清单(已归档)| `data/<用户ID>/list/completed.json`                                    |
| 倒数日     | `data/<用户ID>/countdown/categories.json`、`countdowns.json`            |
| 课程表     | `data/<用户ID>/course/courses.json`                                     |
| 我的       | `data/<用户ID>/profile/profile.json`                                    |
| 会话       | `data/<用户ID>/session/session.json`                                    |
| 设置       | `data/<用户ID>/settings/settings.json`                                  |
| 系统状态   | `data/<用户ID>/system/state.json`                                       |

#### 数据服务层架构

```
┌─────────────────────────────────────┐
│           组件 / Store              │
├─────────────────────────────────────┤
│        storageService.ts            │  ← 统一读写层，内置缓存 + 请求去重
├─────────────────────────────────────┤
│  api.ts  ←→  fileStore.ts           │  ← 双平台自适应
├─────────────────────────────────────┤
│  Express 后端  │  Capacitor FS     │  ← 物理存储
└─────────────────────────────────────┘
```

- **`api.ts`**：认证和业务数据的 HTTP API 调用，自动适配双平台
- **`storageService.ts`**：模块数据的统一读写层，内置 `Map` 缓存和请求去重（`pendingRequests`）
- **`fileStore.ts`**：Capacitor 模式下的本地文件存储实现，路径结构与 Express 后端完全一致
- **`prod-server.cjs`** 内含数据目录结构注释，列出了所有支持的数据类型：`profile`、`session`、`settings`、`footprint`、`list`、`course`、`focus`、`countdown`、`system`

#### 数据加载规则

- 程序启动后，`App.vue` 在 `onMounted` 中调用 `initializeData()`，统一加载所有模块数据
- 数据加载完成后，自动检查并创建新用户默认数据（默认清单、示例任务、示例课程、示例足迹、示例日记），通过 `state.json` 中的 `defaultsInitialized` 字段防止重复创建
- 各 Store 提供 `loadData()` 方法，内置 `isLoaded` 标记防止重复加载
- 功能页面切换时**不重复加载**数据
- 提供 `handleRefreshData()` 手动刷新机制：`clearCache()` + `initializeData()`

### 5.3 script setup 声明顺序

在 `<script setup lang="ts">` 中，所有 `const`/`let` 声明按代码顺序执行且存在临时死区（Temporal Dead Zone, TDZ），引用尚未初始化的变量会抛出 `ReferenceError`。必须遵守以下顺序规则：

1. **所有 `ref`、`computed`、`reactive` 声明集中在 setup 前半部分**
2. **所有函数声明紧随其后**
3. **`watch` 和 `watchEffect` 必须放在所有被引用的数据声明和函数声明之后**

> **原因**：带有 `immediate: true` 的 watch 在创建时立即执行回调，如果回调或依赖数组中引用了后续声明的变量，会触发 TDZ 错误。即使不带 `immediate` 的 watch，其依赖数组在创建时也会被求值。


| 级别    | 方法                 | 用途                         |
| ------- | -------------------- | ---------------------------- |
| TRACE   | `logger.trace()`     | 最细粒度的追踪信息            |
| DEBUG   | `logger.debug()`     | 调试信息（开发环境）          |
| INFO    | `logger.info()`      | 常规操作记录                 |
| WARN    | `logger.warn()`      | 警告信息                     |
| ERROR   | `logger.error()`     | 错误信息（自动附带堆栈）      |

**使用规范**：

- 日志自动上传到服务端（`/api/logs`），同时输出到浏览器控制台，并存入内存缓冲区（最多5000条）
- 在安卓端，日志还会通过 `Filesystem.writeFile` 将完整内存日志写入本地磁盘文件 `{Data}/logs/app-YYYY-MM-DD.log`，使得文件管理器可以浏览日志文件，与电脑版体验一致。写入采用"1 秒防抖合并 + 失败自动重试"策略（防抖定时器合并短时间内的多次写入请求，减少 Capacitor 原生调用频率；每次写入完整内存缓冲区；先尝试写文件，失败时自动创建 `logs` 目录后重试），无需维护写入状态标志或目录就绪标志，避免状态不同步问题。首次写入时自动读取磁盘上已有的历史日志，与当前缓冲区内容合并后写入，确保重启软件后历史日志不丢失
- 日志格式：`[时间戳] LEVEL: 消息 + meta 对象`
- 时间戳使用北京时间（`Asia/Shanghai`）
- 可通过 `logger.getLogs()` 获取内存缓冲区中的日志行数组，供插件等组件读取
- **关键操作必须添加日志**：路由切换、数据操作、登出、错误
- 日志上传为静默失败，不影响用户体验
- 以下模块已添加完整的 `logger.debug` 日志覆盖：
  - 清单页面（`ListPage.vue`）
  - 导航状态管理（`usePageNav.ts`）
  - 清单数据存储（`listStore.ts`）
  - 倒数日页面（`CountdownList.vue`）
  - 足迹页面（`TaskList.vue`）
  - 应用入口（`App.vue`）

### 6.4 导航状态管理

#### 核心接口

```typescript
// 导航上下文
interface NavContext {
  segments: BreadcrumbSegment[]
  plusVisible: boolean
  plusOnClick: (() => void) | null
  goModuleHome: () => void
  actions?: NavAction[]
}

// 面包屑分段
interface BreadcrumbSegment {
  label: string
  color: string
  clickable: boolean
  onClick: (() => void) | null
  dropdownItems: DropdownItem[] | null
}

// 下拉项
interface DropdownItem {
  id: string
  name: string
  color: string
  current: boolean
  onSelect: () => void
}

// 操作按钮
interface NavAction {
  icon: string        // emoji 字符
  onClick: () => void
  title: string
}
```

#### 使用方式

- 子页面通过 `usePageNav()` 获取 `pageNav` 实例
- 使用 `pageNav.navPath`（computed）读取路径
- 使用 `pageNav.setNavPath()` 更新路径
- 清单和倒数日页面通过 `watch` 监听 navPath 在本地计算面包屑分段并渲染

#### 状态持久化

- 清单模块 navPath 通过 `setSystemStateField('list', { navPath })` 持久化
- 倒数日模块 navPath 通过 `setSystemStateField('countdown', { navPath })` 持久化
- navPath **全深度持久化**（包括根级路径），使用 300ms 防抖延迟，确保模块内页面恢复完成后才写入存储
- 恢复时 `restoreModuleNavPath` 严格验证路径首元素必须匹配当前模块名
- 倒数日模块在 `initFilterState` 中：当 navPath 仅为 `['countdown']` 时自动从持久化存储恢复上次浏览的分类页面；当 navPath 已有具体分类路径时跳过恢复
- 旧版导航数据自动迁移为新版格式

#### el-scrollbar 容器布局规则

```css
/* 父容器 */
.parent {
  flex: 1;
  min-height: 0;
}

/* el-scrollbar */
:deep(.el-scrollbar) { height: 100%; }
:deep(.el-scrollbar__view) { min-height: 100%; }
```

- 内容区 padding 设置在 `:deep(.el-scrollbar__view)` 内部
- **避免**在 `el-scrollbar` 根元素上设置 `overflow: hidden` 或 `flex: 1`

## 十一、插件系统

### 11.1 概述

插件系统允许扩展 Earth Survival Diary 的功能，支持三种扩展方式：

1. **页面覆盖**：用同名组件覆盖任意模块的默认页面
2. **Store 覆盖**：用自定义 Store 文件替换默认的数据持久化逻辑
3. **小工具开发**：开发独立小工具，自动展示在工具箱页面

系统启动时通过 [pluginLoader.ts](file:///c:/Project/earth-survival-diary/src/lib/pluginLoader.ts) 自动扫描 `src/plugins/` 目录下的所有插件。

### 11.2 插件目录结构

```
src/plugins/<插件目录名>/
├── plugin.json        # 插件清单（必需）
├── index.ts           # 插件入口（必需）
├── pages/             # 页面覆盖（可选）
│   └── <组件名>.vue
├── stores/            # 自定义 Store（可选）
│   └── <任意名>.ts
└── tools/             # 小工具（可选）
    └── <工具目录名>/
        └── index.vue
```

**`index.ts` 默认导出格式**：
```typescript
export default {
  manifest,   // plugin.json 导入的对象
  pages?,     // 可选：{ 组件名: () => import('./pages/组件名.vue') }
  stores?,    // 可选：{ 文件名: () => import('./stores/文件名.ts') }
  tools?,     // 可选：{ 工具目录名: () => import('./tools/工具目录名/index.vue') }
}
```

### 11.3 插件清单格式

```json
{
  "id": "my-plugin",
  "name": "我的插件",
  "version": "1.0.0",
  "description": "插件描述",
  "author": "开发者名称",
  "tools": {
    "工具目录名": {
      "name": "工具显示名称",
      "description": "工具功能描述",
      "icon": "🔧"
    }
  }
}
```

| 字段          | 必需 | 说明                            |
| ------------- | ---- | ------------------------------- |
| `id`          | 是   | 唯一标识，建议英文短横线命名     |
| `name`        | 是   | 展示名称                        |
| `version`     | 是   | 语义化版本号                    |
| `description` | 是   | 功能简述                        |
| `author`      | 是   | 开发者署名                      |
| `tools`       | 否   | 小工具元数据，key 为工具目录名    |

### 11.4 页面覆盖规则

- 插件中的页面组件名必须与默认组件名**完全一致**
- 组件需放在 `pages/` 目录下，文件名格式为 `<组件名>.vue`
- 当多个插件都提供同一页面覆盖时，按扫描顺序取第一个
- 被覆盖的页面由 `defineAsyncComponent` 异步加载

页面组件名与路由的对应关系：

| `profile`     | `ProfilePage`     | `src/components/profile/ProfilePage.vue`      |

### 11.5 Store 覆盖规则

- 开发者可在 `stores/` 目录下创建**任意名称**的 `.ts` 文件
- Store 文件使用 Pinia Setup Store 风格编写
- 所有插件的 Store 通过 `getAllPluginStores()` 获取，key 为文件名（不含扩展名）

### 11.6 小工具开发规则

- 每个小工具放在 `tools/<工具目录名>/` 下，入口文件为 `index.vue`
- 工具元数据在 `plugin.json` 的 `tools` 字段中定义
- 使用 Vue 3 `<script setup lang="ts">` 语法，样式使用 `<style scoped>`
- 小工具展示在工具箱页面，点击后以新页面形式覆盖工具箱页面
- 新页面宽度为界面宽度的 80%，顶部提供返回图标胶囊按钮，点击返回工具列表
- 新页面内容区可滚动，滚动条半透明隐藏式

### 11.7 开发步骤

1. 在 `src/plugins/` 下创建插件目录（如 `my-plugin/`）
2. 创建 `plugin.json` 填写插件信息
3. 创建 `index.ts` 入口文件，导出 `{ manifest, pages?, stores?, tools? }`
4. 根据需要创建 `pages/`、`stores/`、`tools/` 子目录
5. 编写覆盖页面 / Store / 小工具代码
6. 运行 `pnpm build` 构建验证

## 十三、提醒系统

### 13.1 架构概述

提醒系统支持两种运行模式：

**Electron 桌面端**：提醒由 Electron 主进程管理，前端提交调度，主进程负责定时触发：

```
前端收集提醒数据 → scheduleReminders() → 主进程 timer 队列 → 到期触发 show-reminder → 页面顶部居中弹窗显示 + 提示音
```

1. 前端收集所有需要提醒的任务/倒数日事件
2. 调用 `window.electronAPI.scheduleReminders(reminders, persistDuration)` 提交调度
3. 主进程维护 `reminderTimers` 数组和 `reminderQueue` 队列
4. 触发时通过 `show-reminder` IPC 事件发送提醒到渲染进程
5. 渲染进程在页面顶部水平居中位置显示弹窗，播放提示音（最新弹窗完全重叠在最上方）

**Android (Capacitor) 端**：通过 JS 定时器在应用内提醒（不使用系统通知）：

```
前端收集提醒数据 → 清理旧定时器 → setTimeout() 队列 → 到期推入 activeReminders → 页面顶部居中弹窗显示 + 提示音
```

1. 前端收集所有需要提醒的任务/倒数日事件（与桌面端共用数据收集逻辑）
2. 清理上轮所有 JS 定时器
3. 对每个提醒创建 `setTimeout()` 定时器，到期时将提醒推入 `activeReminders` 数组并播放提示音
4. 提醒弹窗与桌面端保持一致——页面顶部水平居中，最新弹窗完全重叠在最上方
5. 最多调度 7 天内的提醒（`MAX_REMINDER_DELAY`），超过此范围等下次启动重新调度

**注意**：Android 端提醒调度必须在 `initializeData()`（加载所有 store 数据）完成之后进行，否则 store 数据为空会导致收集不到任何提醒。
**注意**：登出时需清理所有 JS 定时器和 Electron 主进程提醒队列，防止残留提醒。
**注意**：正计时模式每满 1 小时会触发一次「focus-hourly」专注提醒（id 为 `focus-hourly`），该提醒使用 `hourly` 重复策略自动调度下一轮。在 FocusTimer.vue 开始/取消/完成正计时时会调用 `refreshReminders()` 以更新提醒调度。

### 13.2 提醒数据格式

```typescript
interface ReminderItem {
  id: string
  name: string
  body: string
  triggerTime: string          // ISO 格式触发时间
  listName?: string
  groupName?: string
  folderName?: string
  repeatStrategy?: string
  courseStartTime?: string     // 课程提醒专用：实际上课时间（ISO 格式，不含时区）
}
```

### 13.3 重复调度规则

- 支持的重复策略：`daily` / `weekly` / `weekly_select` / `monthly` / `monthly_selected_day` / `monthly_last_day` / `yearly` / `custom_days` / `weekdays` / `lunar_date`
- `weekly_select` 策略将选中的星期几（`repeatWeekdays: number[]`，0=周一）持久化到数据库，任务完成时通过 `getNextRepeatDate` 计算下一个选中的星期几
- `monthly_selected_day` 策略将指定的日期数字（`repeatMonthDay: number`，1~31）持久化到数据库，任务完成时计算下个月是否有该日期，若不存在则自动顺延到有该日期的月份
- `lunar_date` 策略将农历月（`repeatLunarMonth: number`，1~12）和农历日（`repeatLunarDay: number`，1~30）持久化到数据库，添加/编辑时农历日通过下拉列表选择（初一~三十），任务卡片上显示"每年正月廿三重复"这样的汉字格式（月和日均使用汉字名称），任务完成时通过 `lunar-javascript` 库计算下一个公历日期，若当前农历年已过则顺延至下一个农历年
- 最大调度时间窗口：20 天（`MAX_SCHEDULE_DELAY`），超过此范围的等下次启动重新调度；Android 端 JS 定时器限制为 7 天
- 登出时调用 `cancelAllReminders()`（Electron）并清理所有 JS 定时器（Android），防止残留提醒
- 每年重复的倒数日在过期后次年仍能正常提醒（通过 `getNextOccurrence` 滚动至下一个未过期年份）
- 用户修改日期、切换重复策略或修改提醒设置后，自动触发 `refreshReminders()` 重新调度
